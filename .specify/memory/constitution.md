# Todo Evolution Project Constitution

## Core Principles

### I. Spec-Driven Development Only

Every feature MUST be defined in a written Markdown Specification before implementation. No implementation may begin without an approved spec.

**Non-Negotiable Rules**:
- All code MUST be generated by **Claude Code**
- Manual code writing, editing, or patching is **strictly forbidden**
- If generated code is incorrect, the spec MUST be refined, not the code
- No exceptions to the spec-first requirement

**Rationale**: Enforcing specification-first development ensures traceability, reproducibility, and eliminates drift between intent and implementation.

### II. Incremental Evolution

The project evolves sequentially across five phases. Each phase builds on the previous one without breaking existing functionality.

**Non-Negotiable Rules**:
- Phases MUST be completed in order (I → II → III → IV → V)
- Backward compatibility MUST be preserved unless explicitly stated in the spec
- Phase completion requires all acceptance criteria met
- No skipping phases or parallel phase development

**Rationale**: Sequential evolution reduces complexity, enables proper testing at each stage, and ensures a solid foundation before adding advanced features.

### III. Single Source of Truth

The Markdown specifications and this Constitution are the authoritative source for system behavior, business logic, API contracts, and infrastructure design.

**Non-Negotiable Rules**:
- If code behavior differs from the spec, the spec prevails
- All team members MUST reference specs, not code, for requirements
- Specs MUST be updated before code when requirements change
- Constitution takes precedence over individual feature specs

**Rationale**: A single source of truth eliminates ambiguity, reduces communication overhead, and provides clear audit trails for decisions.

## Phase Structure & Scope

### Phase I – In-Memory Console Application

**Technology Stack**:
- Python
- Claude Code
- Spec-Kit Plus

**Scope**:
- In-memory task storage
- Console-based interaction
- Core Todo functionality only

**Completion Criteria**: Console application runs with CRUD operations on in-memory tasks.

### Phase II – Full-Stack Web Application

**Technology Stack**:
- Next.js (Frontend)
- FastAPI (Backend)
- SQLModel
- Neon Database (PostgreSQL)

**Scope**:
- Persistent task storage
- REST API
- Web UI for task management
- Feature parity with Phase I plus usability enhancements

**Completion Criteria**: Web application with persistent storage, full CRUD via REST API, and functional UI.

### Phase III – AI-Powered Todo Chatbot

**Technology Stack**:
- OpenAI ChatKit (Chat UI)
- OpenAI Agents SDK (orchestration framework)
- Google Gemini API & Model (via OpenAI-compatible endpoint)

**Scope**:
- Conversational interface for task management
- Natural language commands (e.g., scheduling, updating, filtering tasks)
- AI agent orchestrating Todo operations via APIs

**Completion Criteria**: Chatbot interprets natural language and performs CRUD operations through API.

### Phase IV – Local Kubernetes Deployment

**Technology Stack**:
- Docker
- Minikube
- Helm
- kubectl-ai
- kagent

**Scope**:
- Containerized services
- Local Kubernetes deployment
- AI chatbot running inside the cluster
- Infrastructure defined via Helm charts

**Completion Criteria**: All services containerized and running on local Kubernetes with Helm-managed deployment.

### Phase V – Advanced Cloud Deployment

**Technology Stack**:
- DigitalOcean Kubernetes (DOKS)
- Kafka
- Dapr

**Scope**:
- Cloud-native deployment
- Event-driven architecture
- Scalable AI chatbot and Todo services
- Production-grade observability and resilience

**Completion Criteria**: Production deployment on DOKS with event-driven architecture and observability.

## Todo Feature Evolution

### Basic Features (Phase I+)

- Add Task
- Delete Task
- Update Task
- View Task List
- Mark Task as Complete

### Intermediate Features (Phase II+)

- Priorities (High / Medium / Low)
- Tags / Categories
- Search by keyword
- Filter by status, priority, or date
- Sort by due date, priority, or name

### Advanced Features (Phase III+)

- Recurring tasks
- Due dates and deadlines
- Time-based reminders
- Intelligent rescheduling via AI chatbot

## AI Interaction Rules

- The chatbot MUST translate natural language into structured Todo operations
- The AI MUST NOT directly manipulate storage layers
- All actions MUST flow through:
  - Defined APIs
  - Agent tools
  - MCP-compliant interfaces
- Ambiguous user input MUST trigger clarification requests
- AI MUST NOT assume intent when input is unclear

## Deployment & Infrastructure Rules

- Infrastructure MUST be defined declaratively
- No manual configuration inside running containers or clusters
- Environment parity MUST be maintained between:
  - Local (Minikube)
  - Cloud (DOKS)
- All infrastructure changes MUST go through version-controlled definitions
- Secrets MUST be managed via environment variables or secret managers, never hardcoded

## Documentation Requirements

For every phase, the following MUST exist:

1. Markdown Constitution (this document evolves if needed)
2. Feature Specifications
3. Architecture Overview
4. API Contracts (where applicable)
5. Deployment Specifications (Phases IV & V)

All documentation MUST be maintained in sync with implementation.

## Compliance & Validation

A phase is considered complete ONLY when:

- All features are implemented via Claude Code
- Generated code matches the approved spec
- The application runs successfully in its target environment
- All acceptance criteria pass verification

Any violation of spec-driven rules invalidates the phase.

## Amendment Policy

This Constitution may only be amended via:

- A new Markdown revision
- Explicit justification for the change
- Version increment following semantic versioning

Amendments MUST NOT retroactively weaken constraints.

## Final Declaration

This Constitution enforces discipline, reproducibility, and correctness throughout the Evolution of Todo Project. All contributors agree to follow its rules without exception.

## Governance

- This Constitution supersedes all other practices and ad-hoc decisions
- Amendments require: documentation, explicit justification, and migration plan if breaking
- All PRs/reviews MUST verify compliance with this Constitution
- Complexity beyond what this Constitution permits MUST be justified in writing
- Use CLAUDE.md for runtime development guidance
